State

Two springs, not one store.

React Query owns everything that came from the backend — it's the data layer and the single source of truth. Zustand holds only what's ephemeral: the search term and which cell is playing. Components read the data straight from the query (filtered in select); the store never gets a copy. Two independent sources, flowing down and branching out — like water down a river.

Data flow
React Query · data Zustand · UI state
At scale I'd reach for MobX — computed + fine-grained reactivity kill the derive-with-hooks trap. Zustand won here on 1 KB and no magic: readable in 30 seconds, which is exactly what a reviewer needs. — the state call, in one breath
Virtualization

Five thousand files. Eight rows.

Mounting a <video> for every file would choke the DOM. So only the rows inside the viewport actually exist — the rest is empty space the browser scrolls past. Scroll the strip: the list is enormous, the mounted count barely moves.

0 / 5,000 ROWS MOUNTED IN THE DOM
0% SCROLLED · 0–0 visible

Every row you don't see is one the browser never rendered. That's the whole trick — the DOM stays tiny no matter how big the list.

Testing

Test the logic. Smoke-test the paint.

Confidence comes from deterministic logic and state tests — pass data in, assert the output, no mocked network. The one render test proves the wiring, nothing more: it can't even mount the grid without faking ResizeObserver, and a test that fakes the browser isn't testing the browser. Real UI confidence lives in E2E, in a real engine.

Aside

The throne was never filled.

React never crowned a state leader. Redux got dethroned by its own creator; Recoil shipped, got hyped, then quietly went dark; Zustand, Jotai and Valtio are — funnily enough — the same three people. Pick the one that fits the job, not the sermon. (Hover the dots.)

The rest of the record

Four smaller calls.

Styling

Pure CSS, no Tailwind.

Nesting and custom properties ship in the browser now. A utility framework just makes the HTML — and the selectors — uglier as the app grows.

Playback

The vault:// protocol.

The renderer can't touch file://. So the main process serves a custom scheme — streaming the real file with HTTP Range support, so every cell plays a real, seekable video.

<video>→ vault://→ main→ stream + Range
Fetching

One query per cell.

Not one big request. A single failed lookup shows a Retry on that card — the other 4,999 don't care. The mock fails ~10% of the time to prove it.

Mock API

Per-path, not batch.

The brief says array → list. I went getFileMetadata(path) on purpose — it's the only shape that makes the per-cell loading / error / retry states above possible.
